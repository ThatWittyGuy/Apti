Building a Collaborative Document Editor: Real-Time Synchronization, CRDTs, and Conflict Resolution

How does Google Docs allow thousands of users to edit the same document simultaneously without conflicts? When you type a character, it…
Arvind Kumar
a11y-light · December 2, 2025 (Updated: December 4, 2025) · Free: No

How does Google Docs allow thousands of users to edit the same document simultaneously without conflicts? When you type a character, it appears on everyone's screen in real-time. But what happens when two people type at the same position? Or delete the same word? Let's design a collaborative editor that handles it all.

A Real-World Problem

Aadvik (Interviewer): "Sara, imagine you're building Google Docs. Multiple users are editing the same document in real-time. User A types 'Hello' at position 0. User B types 'World' at position 5. Meanwhile, User C deletes characters from position 2–4. How do you ensure everyone sees the same final document? How do you handle conflicts when operations arrive out of order?"

Sara (Candidate): [Thoughtful pause] "This is the classic collaborative editing problem! In a single-user editor, it's simple — just apply operations sequentially. But with multiple concurrent editors, operations can arrive in different orders, and we need to ensure all replicas converge to the same state. This requires conflict-free synchronization algorithms."

Aadvik: "Exactly. Traditional version control (like Git) won't work — it's too slow and requires manual merges. We need algorithms that automatically resolve conflicts. Ready to design a real-time collaborative editor?"

Sara: "Absolutely. Let me start with some clarifying questions:

  • What's the scale? How many concurrent editors per document?
  • What operations do we need to support? Insert, delete, formatting?
  • What's the acceptable latency for seeing others' changes?
  • Do we need offline editing support?
  • How do we handle version history and undo/redo?
  • What about permissions and access control?"

Aadvik: "Great questions! Let's define the requirements:

Part 1: Requirements, Core Challenges & Scale Estimation

1. Functional Requirements:

  1. Create, edit, and delete documents
  2. Real-time collaborative editing (multiple users simultaneously)
  3. Text operations: insert, delete, format (bold, italic)
  4. Version history and undo/redo
  5. Document sharing and permissions
  6. Presence awareness (show who's editing where)

2. Non-Functional Requirements:

  1. Scale: 1000 concurrent editors per document (peak: 5000)
  2. Latency: <100ms for changes to appear on other clients
  3. Availability: 99.9% (allows ~43 minutes downtime/month)
  4. Consistency: Eventually consistent (all replicas converge to same state)
  5. Conflict resolution: Automatic, no manual merges
  6. Offline support: Allow editing offline, sync when online

3. Core Challenges:

  1. Concurrent Edits: Multiple users editing simultaneously
  2. Operation Ordering: Operations arrive in different orders on different clients
  3. Conflict Resolution: How to merge concurrent edits automatically
  4. Causality: Ensuring operations that depend on each other are applied correctly
  5. Scalability: Handling thousands of concurrent editors efficiently
  6. Global Scale: Users across different regions need low latency
  7. Cross-Region Sync: Ensuring consistency across geographic regions"

Aadvik: "One more thing — this needs to work globally. Users in New York, London, and Tokyo should all have a responsive experience. Network latency between regions can be 200–300ms. How do you keep it snappy?"

Sara: "That's a critical requirement. We'll need per-region routing so users connect to their nearest region, with intelligent cross-region replication. The CRDT approach makes this much simpler since operations are commutative."

Aadvik: "Let's start by understanding why this is hard. In a single-user editor, it's simple:"

// Simple case: one user
document = "Hello"
document.insert(5, " World")  // "Hello World"
document.delete(0, 5)         // " World"
// Final: " World"

"But with multiple users editing simultaneously:"

// User A: Inserts "Hello" at position 0
// User B: Inserts "World" at position 5 (they see "Hello")
// User C: Deletes position 2-4 (they see "Hello")

// All operations happen concurrently
// Different clients see operations in different orders
// How do we ensure everyone gets "Hlo World" or "Hel World"?

Sara: "The fundamental challenge is causality preservation and convergence. We need:

  1. Causality: If operation A happened before B, all clients must apply A before B
  2. Convergence: All clients eventually see the same document state
  3. Intention Preservation: The result should match what users intended

There are two main approaches: Operational Transform (OT) and Conflict-free Replicated Data Types (CRDTs)."

4. Scale Estimation

Aadvik: "Before we dive into architecture, let's estimate the scale. This helps us understand the infrastructure requirements."

Sara: "Let me break down the numbers:

a. Traffic Estimation:

For 1 million concurrent users (across all documents):

  • Average typing rate: 5 characters per second per user
  • Peak typing rate: 10 characters per second (fast typers)
  • Total edit operations: 1M users × 5 ops/sec = 5 million operations/second
  • Peak load: 1M × 10 = 10 million operations/second

b. Document Operations:

  • 100,000 documents being edited concurrently
  • Average 10 users per document = 1M users
  • Each edit operation generates:
  • 1 WebSocket message to server
  • Server broadcasts to ~10 other users in same document
  • Total messages: 5M ops/sec × 10 = 50 million messages/second

c. Storage Requirements:

Document State:

  • Average document: 50,000 characters
  • With metadata (formatting, structure): ~100KB per document
  • 100,000 active documents: 100K × 100KB = 10 GB (in-memory)

Operation Log (for conflict resolution):

  • Each operation: ~200 bytes (position, content, user_id, timestamp, vector clock)
  • 5M ops/sec × 200 bytes = 1 GB/second write rate
  • Keep last 1 hour of operations: 1 GB/sec × 3600 = 3.6 TB per hour
  • With compression: ~1 TB per hour

Presence Data:

  • Per user presence: ~100 bytes (user_id, cursor position, selection)
  • 1M concurrent users: 1M × 100 bytes = 100 MB (in-memory)

d. Bandwidth:

Inbound (Client → Server):

  • 5M ops/sec × 200 bytes = 1 Gbps

Outbound (Server → Clients):

  • 50M messages/sec × 200 bytes = 10 Gbps

e. Database Requirements:

Document Metadata:

  • Document ID, title, owner, permissions, last modified
  • ~500 bytes per document
  • 100,000 active documents: 50 MB

Operation History (for undo/redo):

  • Store operations for last 24 hours
  • 5M ops/sec × 24 hours × 200 bytes = 86.4 TB per day
  • With compression and archiving: ~20 TB per day

Summary:

Edit operations: 5M/sec (peak: 10M/sec)

Messages: 50M/sec

Bandwidth: 11 Gbps total

Storage: ~1 TB/hour for operation log

Memory: ~10 GB for active document state"

Part 3: CRDTs vs Operational Transform

Aadvik: "You mentioned two approaches: CRDTs and OT. Explain both."

Sara: "Both solve the same problem but with different strategies:"

Approach 1: Operational Transform (OT)

How it works:

  • Operations are transformed before applying to handle conflicts
  • Each operation is transformed based on operations that happened before it
  • Requires a central server to maintain operation history

Example:

// Initial: "Hello"
// User A: Insert "X" at position 2 → "HeXllo"
// User B: Insert "Y" at position 4 → "HellYo"
// Operations arrive out of order on User B's client:
// B sees: [Insert Y@4] then [Insert X@2]
// Need to transform: T(Insert X@2, Insert Y@4)
// Result: Insert X@2, but Y is now at position 5
// Final: "HeXllYo"

Pros:

  • Smaller message size (operations, not full state)
  • Familiar to developers
  • Used by Google Docs (early versions)

Cons:

  • Complex transformation rules
  • Requires central server (less distributed)
  • Hard to implement correctly
  • Difficult to handle all edge cases

Approach 2: Conflict-free Replicated Data Types (CRDTs)

How it works:

  • Data structure designed to be commutative and associative
  • Operations always merge correctly regardless of order
  • True distributed (no central server needed)

Example with Character-based CRDT:

// Each character has unique ID (site ID + logical clock)
// Characters are stored in a sequence that preserves order

// Initial: []
// User A (site 1): Insert "H" → [{id: 1:1, char: 'H', pos: 0}]
// User B (site 2): Insert "W" → [{id: 2:1, char: 'W', pos: 0}]
// Merge (both operations):
// Sort by (site, clock) → [{id: 1:1, char: 'H'}, {id: 2:1, char: 'W'}]
// Display: "HW"
// If User A then inserts "e" after "H":
// [{id: 1:1, char: 'H'}, {id: 1:2, char: 'e', after: 1:1}, {id: 2:1, char: 'W'}]
// Display: "HeW"

Pros:

  • Simple merge (just union of sets)
  • Truly distributed (no central server)
  • Mathematically proven to converge
  • Easier to implement correctly

Cons:

  • Larger message size (unique IDs for each character)
  • Requires garbage collection for deleted characters
  • Memory overhead for tombstone markers

Aadvik: "Which approach would you choose?"

Sara: "For a modern system, I'd recommend CRDTs because:

  1. Simpler to implement correctly
  2. Truly distributed (better scalability)
  3. Mathematically sound (proven convergence)
  4. Works offline (no central server needed)

Google Docs now uses a hybrid approach, but many modern editors (like Notion, Figma) use CRDTs."

Part 4: CRDT Implementation Deep Dive

Aadvik: "Let's dive into CRDT implementation. How does it actually work?"

Sara: "For text editing, we use a Logoot-like CRDT (or RGA — Replicated Growable Array). Here's how:"

1. Character-Based CRDT

Core Idea:

  • Each character has a unique identifier (site ID + logical timestamp)
  • Characters form a sequence that preserves causality
  • Deletions are marked with tombstones (not immediately removed)

Data Structure:

// Character with unique identifier
public class Char {
    String id;        // "site_id:logical_timestamp"
    char value;       // The actual character
    String prevId;    // Previous character ID (for ordering)
    boolean deleted;  // Tombstone marker
}

// Document as ordered list of characters
List<Char> document = [
    {id: "1:1", value: 'H', prevId: null, deleted: false},
    {id: "1:2", value: 'e', prevId: "1:1", deleted: false},
    {id: "2:1", value: 'W', prevId: "1:1", deleted: false}
];

Insert Operation:

// User inserts 'X' after character with id "1:2"
public void insert(char value, String afterId, String siteId) {
    String newId = generateId(siteId);  // "1:3"
    
    Char newChar = new Char(
        id: newId,
        value: value,
        prevId: afterId,
        deleted: false
    );
    
    // Add to local document
    document.add(newChar);
    
    // Broadcast operation
    broadcast({
        type: "INSERT",
        char: newChar,
        afterId: afterId
    });
}

Merge Operation (Receive from other client):

// Receive insert operation from another client
public void mergeInsert(Char newChar) {
    // Find insertion point (character with prevId)
    int insertPos = findInsertionPoint(newChar.prevId);
    
    // Insert in correct position (maintain order by ID)
    document.add(insertPos, newChar);
    
    // Sort by ID to maintain causality
    document.sort((a, b) -> compareIds(a.id, b.id));
    
    // Update UI
    updateDisplay();
}

Delete Operation:

// User deletes character with id "1:2"
public void delete(String charId) {
    // Mark as deleted (tombstone)
    Char charToDelete = findById(charId);
    charToDelete.deleted = true;
    
    // Broadcast delete operation
    broadcast({
        type: "DELETE",
        charId: charId
    });
    
    // Update UI (hide deleted character)
    updateDisplay();
}

2. Version Vectors for Causality

Aadvik: "How do you ensure causality? What if operations depend on each other?"

Sara: "We use version vectors (also called vector clocks) to track which operations each site has seen:"

// Version vector: map of site_id -> logical_timestamp
Map<String, Long> versionVector = {
    "site_1": 5,  // Site 1 has seen 5 operations from site_1
    "site_2": 3,  // Site 1 has seen 3 operations from site_2
    "site_3": 7   // Site 1 has seen 7 operations from site_3
};

// When sending operation:
public Operation createOperation(char value, String afterId) {
    String siteId = getSiteId();
    Long timestamp = incrementLocalClock(siteId);
    
    return {
        type: "INSERT",
        char: {id: siteId + ":" + timestamp, value: value, prevId: afterId},
        versionVector: getCurrentVersionVector()  // Include current state
    };
}
// When receiving operation:
public void receiveOperation(Operation op) {
    // Check if we've already seen dependencies
    if (hasSeenAllDependencies(op.versionVector)) {
        // Safe to apply
        mergeOperation(op);
        updateVersionVector(op.versionVector);
    } else {
        // Wait for dependencies (queue operation)
        queueOperation(op);
    }
}

Part 5: Real-Time Synchronization Flow

Aadvik: "Show me the complete flow when multiple users are editing."

Sara: "Here's the end-to-end flow with CRDT:"

1. Complete Synchronization Flow

Key Points:

  1. Optimistic Updates: Users see their changes immediately (no waiting for server)
  2. CRDT Merge: Operations merge automatically regardless of order
  3. Version Vectors: Ensure causality (operations applied in correct order)
  4. Eventual Consistency: All clients converge to same state

2. Per-Region Routing & Cross-Region Replication

Aadvik: "You mentioned multi-region support. How does per-region routing work? What happens when users in different regions edit the same document?"

Sara: "Per-region routing is critical for low latency. Here's how it works:"

3. Per-Region Routing Strategy

1. User Connection:

  • User connects to nearest region based on geolocation (DNS-based routing)
  • Global Load Balancer routes WebSocket connection to regional WebSocket servers
  • User's operations go to their local region first (optimistic application)

2. Document Ownership:

  • Each document has a 'home region' (region where it was created)
  • Home region maintains authoritative state
  • Other regions maintain replicas with eventual consistency

3. Operation Flow:

Same Region (Fast Path):

User A (US) → US WebSocket Server → US Document Service → US Database
User B (US) → US WebSocket Server → US Document Service → US Database
- Operations applied immediately
- Broadcast to other users in same region via Kafka
- Latency: < 50ms

Cross-Region (Slower Path):

User A (US) → US WebSocket Server → US Document Service → US Database
User B (EU) → EU WebSocket Server → EU Document Service → EU Database
- Operations applied locally first (optimistic)
- Replicated to other regions asynchronously via Replication Service
- Conflict resolution when operations arrive (CRDT handles this automatically)
- Latency: < 100ms (same region), < 500ms (cross-region)

4. Cross-Region Replication Flow

4. Replication Strategy:

Eventual Consistency Model:

  • Operations applied locally immediately (optimistic)
  • Operations replicated to other regions via Kafka/event stream
  • Each region applies operations in order (using vector clocks)
  • Conflicts resolved using CRDT merge algorithm (automatic)

5. Conflict Resolution Example:

Scenario:

Document: "Hello World"
User A (US): Inserts "Hi " at position 0
User B (EU): Inserts "Beautiful " at position 6
Both operations happen simultaneously

Step-by-Step Resolution:

1. Local Application (Optimistic):

Region US:
- Receives A's operation: Insert "Hi " at 0
- Applies immediately: "Hi Hello World"
- Vector clock: {US: 1, EU: 0}

Region EU:
- Receives B's operation: Insert "Beautiful " at 6
- Applies immediately: "Hello Beautiful World"
- Vector clock: {US: 0, EU: 1}

2. Replication:

US → EU: Operation A with vector clock {US: 1, EU: 0}
EU → US: Operation B with vector clock {US: 0, EU: 1}

3. Conflict Resolution (CRDT Merge):

In Region EU:

  • Has: "Hello Beautiful World" with clock {US: 0, EU: 1}
  • Receives: Insert "Hi " at 0 with clock {US: 1, EU: 0}
  • Since clocks are concurrent (neither dominates), merge:
  • Operation A: Insert at position 0
  • Operation B: Insert at position 6, but A inserted 3 chars, so adjust: position 6 + 3 = 9
  • Result: "Hi Hello Beautiful World" ✓

In Region US:

  • Has: "Hi Hello World" with clock {US: 1, EU: 0}
  • Receives: Insert "Beautiful " at 6 with clock {US: 0, EU: 1}
  • Merge:
  • Operation B: Insert at position 6
  • Operation A already inserted 3 chars at 0, so B's position is correct
  • Result: "Hi Hello Beautiful World" ✓

4. Convergence:

  • Both regions end up with: "Hi Hello Beautiful World"
  • Vector clocks converge: {US: 1, EU: 1}
  • State is consistent across all regions

Key Points:

  • Operations are commutative, so order doesn't matter
  • Position adjustments happen automatically (CRDT handles this)
  • No manual conflict resolution needed
  • All regions converge to same state

6. State Synchronization:

Periodic Sync:

  • Every 1 second, regions exchange operation summaries
  • Compare vector clocks to find missing operations
  • Request missing operations from other regions
  • Apply missing operations to catch up

On Document Open:

  • Client requests latest state from nearest region
  • Region checks if it has latest state (via vector clock)
  • If not, fetches missing operations from home region
  • Applies all operations to reconstruct state

Part 6: Database Design

Aadvik: "How do you store documents in the database? What's the schema?"

Sara: "We need to store both the CRDT state and the rendered document:"

1. Documents Table

CREATE TABLE documents (
    document_id VARCHAR(50) PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    owner_id VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    version_vector JSONB NOT NULL,
    -- Version vector: {"site_1": 100, "site_2": 50, ...}
    
    INDEX idx_owner_id (owner_id),
    INDEX idx_updated_at (updated_at)
);

Purpose: Stores document metadata and current version vector state.

2. Document Operations Table (CRDT State)

CREATE TABLE document_operations (
    operation_id BIGSERIAL PRIMARY KEY,
    document_id VARCHAR(50) NOT NULL,
    site_id VARCHAR(50) NOT NULL,
    logical_timestamp BIGINT NOT NULL,
    operation_type VARCHAR(20) NOT NULL,  -- INSERT, DELETE, FORMAT
    character_id VARCHAR(100) NOT NULL,   -- "site_id:timestamp"
    character_value CHAR(1),
    previous_id VARCHAR(100),            -- Previous character ID
    deleted BOOLEAN DEFAULT false,        -- Tombstone for DELETE
    format_data JSONB,                   -- For formatting (bold, italic)
    created_at TIMESTAMP NOT NULL,
    
    FOREIGN KEY (document_id) REFERENCES documents(document_id),
    UNIQUE KEY unique_char_id (document_id, character_id),
    INDEX idx_doc_site (document_id, site_id, logical_timestamp)
);

Purpose: Stores all CRDT operations (characters) that form the document. This is the source of truth.

3. Document Snapshots Table

CREATE TABLE document_snapshots (
    snapshot_id VARCHAR(50) PRIMARY KEY,
    document_id VARCHAR(50) NOT NULL,
    content TEXT NOT NULL,                -- Rendered text (for fast reads)
    version_vector JSONB NOT NULL,
    operation_count INT NOT NULL,         -- Number of operations included
    created_at TIMESTAMP NOT NULL,
    
    FOREIGN KEY (document_id) REFERENCES documents(document_id),
    INDEX idx_doc_created (document_id, created_at DESC)
);

Purpose: Periodic snapshots of rendered document for fast loading (avoid rebuilding from all operations).

4. Document Collaborators Table

CREATE TABLE document_collaborators (
    document_id VARCHAR(50) NOT NULL,
    user_id VARCHAR(50) NOT NULL,
    permission VARCHAR(20) NOT NULL,      -- OWNER, EDITOR, VIEWER
    joined_at TIMESTAMP NOT NULL,
    
    PRIMARY KEY (document_id, user_id),
    INDEX idx_user_id (user_id)
);

Purpose: Tracks who has access to each document and their permissions.

5. Document Sessions Table (Redis)

-- Stored in Redis, not SQL
-- Key: session:doc:{document_id}:user:{user_id}
-- Value: {
--   cursor_position: 10,
--   selection_range: {start: 5, end: 10},
--   version_vector: {...},
--   last_seen: timestamp
-- }
-- TTL: 30 minutes (session expires if inactive)

Purpose: Tracks active editing sessions for presence awareness (who's editing where).

Part 9: Scaling Architecture

Aadvik: "How do you scale to handle 5000 concurrent editors per document?"

Sara: "Scaling strategies:"

Scaling Strategies:

1. WebSocket Server Scaling:

  • Load balancer distributes connections across WebSocket servers
  • Each document can be handled by multiple servers (Kafka for broadcasting)
  • Sticky sessions not required (stateless via Kafka)

2. Kafka Partitioning:

// Partition by document_id for ordered processing per document
kafkaTemplate.send("document-operations-" + documentId, 
                   documentId,  // Partition key
                   operationEvent);
// Same document = same partition = ordered operations

3. Database Sharding:

  • Shard documents by document_id (consistent hashing)
  • Each shard handles subset of documents
  • Operations for same document go to same shard

4. Redis for Presence:

  • Store active sessions in Redis (TTL: 30 minutes)
  • Key: presence:doc:{document_id}:user:{user_id}
  • Expires automatically when user disconnects

5. Operation Batching:

  • Batch multiple operations before sending to Kafka
  • Reduces message overhead
  • Still maintains causality (batch has version vector)

6. Multi-Region Scaling:

  • Each region operates independently
  • Horizontal scaling within each region
  • Cross-region replication via dedicated Replication Service
  • Geographic load distribution (users connect to nearest region)

7. Caching Strategy:

Multi-Layer Caching:

L1: In-Memory (Service Instance)

  • Cache document state for active connections
  • LRU eviction, 100MB per instance
  • Reduces Redis load

L2: Redis (Distributed)

  • Cache document state and presence
  • Shared across all instances in region
  • TTL: 1 hour for state, 60 seconds for presence

L3: Database (Persistent)

  • Source of truth for operations
  • Used for recovery and history

8. Document Sharding:

  • Shard documents by document_id (consistent hashing)
  • Each shard handles subset of documents
  • Operations for same document go to same shard
  • Enables efficient broadcasting (users of same document on same shard)

9. Load Balancing:

  • Sticky sessions for WebSocket (same document → same server when possible)
  • Ensures users of same document on same server (enables efficient broadcasting)
  • Health checks for WebSocket connections
  • Geographic routing (Route53, CloudFlare) for global load balancing

Part 10: Failure Handling & Recovery

Aadvik: "What happens if a client disconnects? How do you recover?"

Sara: "We handle disconnections and recovery:"

Reconnection Flow

Implementation Strategy

1. Operation Logging:

// Store all operations with version vectors
// When client reconnects:
public List<Operation> getMissingOperations(
    String documentId, 
    Map<String, Long> clientVersionVector
) {
    // Find operations client hasn't seen
    return db.query(
        "SELECT * FROM document_operations " +
        "WHERE document_id = ? " +
        "AND (site_id, logical_timestamp) > (client's version vector)",
        documentId
    );
}

2. Snapshot for Fast Recovery:

// Periodic snapshots (every 1000 operations)
// Fast recovery: load snapshot + apply operations after snapshot
public Document recoverDocument(String documentId) {
    Snapshot snapshot = getLatestSnapshot(documentId);
    Document doc = renderFromSnapshot(snapshot);
    
    // Apply operations after snapshot
    List<Operation> ops = getOperationsAfter(snapshot.operationCount);
    for (Operation op : ops) {
        doc.applyOperation(op);
    }
    
    return doc;
}

Additional Failure Scenarios

  1. WebSocket Server Crash:
  • Problem: Service instance dies, WebSocket connections lost
  • Mitigation: - Health checks detect failure - Load balancer stops routing to failed instance - Clients reconnect automatically (exponential backoff) - New instance launched via auto-scaling - Clients fetch latest state on reconnect
  • Recovery: Clients reconnect, state synced from Redis/database

2. Redis Failure:

  • Problem: Document state cache unavailable
  • Mitigation: - Redis Cluster with replicas (automatic failover) - Fallback to database for document state - Reconstruct state from operation log if needed - Cache warming when Redis recovers
  • Impact: Slightly higher latency, but system continues working

3. Database Failure:

  • Problem: Primary database unavailable
  • Mitigation: - Read replicas for querying - Failover to replica (with potential data loss window) - Reconstruct from operation log if needed
  • Recovery: Promote replica to primary, rebuild from operation log

4. Network Issues: Cross-Region Network Partition:

  • Problem: Regions can't communicate, operations can't replicate
  • Mitigation: - Each region continues operating independently - Operations applied locally (optimistic) - Operations queued for replication when connection restored - CRDT ensures no conflicts when operations merge
  • Recovery: When partition heals, regions exchange operations, state converges

5. Data Loss Scenarios: Operation Loss

  • Problem: Operation not persisted before service crash
  • Mitigation: - Write operation to Kafka first (durable) - Then update Redis cache - Kafka ensures at-least-once delivery - Operations are idempotent (can be applied multiple times)
  • Recovery: Replay operations from Kafka if needed

6. State Corruption:

  • Problem: Document state becomes inconsistent
  • Mitigation: - Reconstruct state from operation log - Apply all operations from beginning (or from checkpoint) - Vector clocks ensure correct ordering - Validate state hash after reconstruction

7. Kafka Lag:

  • Problem: Replication falls behind, cross-region sync delayed
  • Mitigation: - Monitor Kafka consumer lag - Scale consumers if lag increases - Increase partition count if needed - Alert if lag exceeds threshold (e.g., 1 second)

8. Consistency Issues: Vector Clock Desynchronization:

  • Problem: Vector clocks get out of sync between regions
  • Mitigation: - Periodic sync: regions exchange vector clocks - Compare clocks to find missing operations - Request missing operations from other regions - Apply missing operations to catch up

9. Operation Ordering:

  • Problem: Operations arrive out of order
  • Mitigation: - CRDT operations are commutative (order doesn't matter) - Vector clocks ensure causal ordering - Operations can be applied in any order, result is same

10. Client-Side Failures: Client Crash:

  • Problem: Client application crashes, unsaved edits lost
  • Mitigation: - Operations sent to server immediately (optimistic) - Server persists operations even if client crashes - On restart, client syncs state from server - All edits are preserved

Part 11: Trade-offs & Final Thoughts

Aadvik: "What are the key trade-offs in this design?"

Sara: "Important trade-offs:

1. CRDT vs OT

  • Chosen: CRDT for simplicity and distributed nature
  • Trade-off: Larger message size (unique IDs per character)
  • Mitigation: Compression, operation batching

2. Optimistic Updates

  • Chosen: Apply operations locally immediately
  • Trade-off: Possible rollback if operation conflicts
  • Mitigation: Rare in practice, UI handles rollback gracefully

3. Eventual Consistency

  • Chosen: Eventually consistent (AP from CAP)
  • Trade-off: Temporary inconsistency during network partitions
  • Mitigation: CRDT guarantees convergence

4. Memory Overhead

  • Chosen: Store all operations (for history and recovery)
  • Trade-off: Database grows over time
  • Mitigation: Periodic snapshots, archive old operations

5. WebSocket Scaling

  • Chosen: Multiple WebSocket servers with Kafka broadcasting
  • Trade-off: More complex architecture
  • Mitigation: Well-established pattern, Kafka handles distribution

6. Per-Region Routing vs Single Region

  • Chosen: Per-region routing for global scale
  • Trade-off: More complex replication, cross-region sync overhead
  • Mitigation: CRDT makes cross-region sync simpler (no conflicts)
  • Benefit: Low latency for users (connect to nearest region)

7. Consistency: Strong vs Eventual

  • Chosen: Eventual consistency (AP from CAP)
  • Trade-off: Users may briefly see different states
  • Mitigation: CRDT guarantees convergence, brief inconsistencies acceptable
  • Benefit: Low latency, works offline, handles partitions

8. Operation-Based vs State-Based

  • Chosen: Operation-based (store operations, not just state)
  • Trade-off: More complex (need to store operations, reconstruct state)
  • Mitigation: Snapshots for fast recovery
  • Benefit: Enables undo/redo, conflict resolution, operation replay

9. In-Memory State vs Database-Only

  • Chosen: Hybrid (In-Memory + Database)
  • Trade-off: More complex (two storage systems, need to keep in sync)
  • Mitigation: Redis for fast access, database for persistence
  • Benefit: Fast access for active documents, persistent storage for operations"

Aadvik: "What would you improve if you had more time?"

Sara: "Additional improvements:

  1. Operation Compression: Compress operation batches before sending
  2. Adaptive Snapshotting: Create snapshots based on document size and activity
  3. Offline Mode: Queue operations locally, sync when online
  4. Conflict Visualization: Show conflict resolution to users
  5. Granular Permissions: Per-section permissions (some users can only edit certain parts)
  6. Rich Formatting CRDT: Extend CRDT to handle formatting (bold, italic, colors)
  7. Performance Optimization: Incremental rendering (only update changed parts of UI)"

Summary

Key Takeaways:

  1. CRDTs provide automatic conflict resolution — Operations merge correctly regardless of order, no manual conflict resolution needed

  4. Event-driven architecture scales — Kafka handles broadcasting to thousands of clients
  5. Per-region routing enables global scale — Users connect to nearest region for low latency
  6. Cross-region replication with eventual consistency — All regions converge to same state automatically
  7. Snapshots enable fast recovery — Rebuild document state quickly after disconnection
  8. Multi-layer caching optimizes performance — In-memory, Redis, and database for different use cases
