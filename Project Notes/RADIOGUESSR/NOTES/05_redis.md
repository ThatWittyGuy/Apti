## What Redis is and how it works here — full explanation

**What Redis is:** Redis (Remote Dictionary Server) is an in-memory data store. It's a separate server process that stores key-value pairs entirely in RAM, making reads and writes sub-millisecond. Unlike a database that writes to disk on every operation, Redis keeps everything in memory and optionally syncs to disk in the background. This makes it extremely fast — typically under 1ms per operation over a local network.

**Why it's different from a JavaScript Map:** A JS Map lives inside your Node.js process. When the process dies (restart, crash, deploy), the Map is gone. Redis is a separate process — your Node server connects to it over TCP (or in Upstash's case, over HTTPS). The data persists independently of your application.

**How Upstash works:** Upstash is Redis-as-a-service over HTTP. Instead of connecting to a Redis server via TCP socket (which requires a native binary client), Upstash exposes a REST API. The `@upstash/redis` client translates Redis commands into HTTP requests: `redis.get('key')` becomes `GET https://xxxx.upstash.io/get/key`. This is why it works on Windows without any native dependencies.

**How TTL works in Redis vs the old Map:** In the old Map, you stored `{ value, expiresAt: Date.now() + ttlMs }` and checked the clock on every read. Redis has TTL as a first-class feature: `SET key value EX 600` tells Redis to automatically delete this key after 600 seconds. You never check the clock. You never see stale data. Redis handles eviction internally using a combination of lazy deletion (on access) and active expiry sweeps. This is more reliable — a key with EX 600 is guaranteed gone at 600 seconds, regardless of whether anyone ever reads it.

**The cache key design:** Keys are structured as `rg:cache:DE:votes:true:40:jazz`. The `rg:cache:` prefix namespaces all cache entries so if you add Redis for other purposes (sessions, rate limiting), their keys don't collide. The rest encodes the exact Radio Browser query — country, sort order, direction, pagination offset, and genre tag. Two requests with different parameters get different cache entries.

**Why cache writes are fire-and-forget here:** After fetching from Radio Browser, the code calls `cache.set(cacheKey, data)` without `await`. This means the server returns the data to the client immediately without waiting for the Redis write to confirm. If the Redis write fails, the cache misses next time but the user still gets their station. This is called a "write-behind" or non-blocking write — correct for a cache where the source of truth is the upstream API, not Redis.

**What happens at scale:** The real power of Redis becomes visible when you run multiple Express instances. With the old Map, instance 1 and instance 2 each had their own cache. If 100 users hit instance 1 for German stations and then a load balancer routes the next 100 to instance 2 — all 100 of those hit Radio Browser again, even though the data was already fetched. With Redis, all instances share one cache. The first instance to fetch `DE:votes:true:40:jazz` populates Redis. Every subsequent instance — regardless of which one handles the request — gets a cache hit.
